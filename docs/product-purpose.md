# 产品目的

## 1. 为什么存在

劳动仲裁材料通常分散在合同、工资记录、考勤、聊天、邮件、录音及用户陈述中。使用 AI 或脚本整理这些材料时，流畅的输出可能掩盖四类问题：材料没有进入分析、引用指向不存在的记录、输入变更后继续复用旧结论，以及未经核验的事实或规则被写成确定结论。

本产品首先解决“自动化过程是否可检查”这个前置问题，再把法律判断交给有资格、有上下文且可追责的外部复核者。

## 2. 长期产品结果

长期目标是建设一套面向中国大陆劳动争议的证据整理、法律核验、请求分析、金额计算、文书生成、庭审准备和风险审查一体化办案系统。第一阶段覆盖北京市、劳动者申请人、劳动仲裁程序和高频争议，之后通过地区规则包扩展。

系统以可复查案件工作区为基础，使复核者能够明确看到：

- 使用了哪些原始材料；
- 每项事实、证据位置、候选规则和计算输入如何关联；
- 哪些材料缺失、冲突或仍未核验；
- 某项输出基于哪个材料、规则和计算版本；
- 输入发生变化后，哪些旧结果必须失效。

在此基础上，系统逐步提供经过版本治理的北京规则、请求权和时效分析、确定性专业计算、文书交付、人工审批及真实案件评测。完整范围、21项能力和验收口径见[最终产品需求基线](final-product-requirements.md)，实施顺序见[最终产品路线图](implementation-roadmap.md)。

长期价值不是代替律师或仲裁员，而是降低遗漏、幻觉、引用漂移、过期法律、错误金额和伪确定性进入正式法律分析及文书的概率。

## 3. v0.3.0 当前任务

当前版本只实现长期链路中的本地技术交接层：

```text
已有本地文件
  -> 有界、只读优先的字节登记
  -> 外部工具或人工创建 v1.3 结构化记录
  -> 引用、清单、快照和通用算术校验
  -> 技术交接包
  -> 项目外部法律复核
```

当前版本不提取文件内容，不判断证据真实性或证明力，不验证法律规则现行有效，不计算仲裁时效或专业请求金额，也不生成或批准提交文书。

## 4. 主要使用时点

在以下时点使用：

1. 材料已经收集，但尚未信赖任何 AI 法律分析或文书；
2. 外部工具或人工已经形成结构化记录，需要检查引用和快照完整性；
3. 原文件、记录、候选规则或计算输入发生改变，需要重新锁定技术状态；
4. 准备把技术底稿交给项目外部法律复核者。

它不是当前面向无技术背景劳动者的一站式办案应用，而是可被后续应用、Agent 或人工流程调用的开源内核。

## 5. 输入、处理与输出

| 阶段 | 当前版本负责 | 当前版本不负责 |
| --- | --- | --- |
| 输入 | 用户指定的本地文件；调用方提供的 v1.3 结构化案件包 | 自动识别完整案件事实或法律问题 |
| 处理 | 有界扫描、字节哈希、Schema、ID/引用、清单、快照和通用加法校验 | OCR、语义判断、现行法律核验、时效和专业金额计算 |
| 输出 | 原始文件清单、锁定案件包、确定性校验报告和未验证范围 | 法律意见、胜诉判断、提交文书或人工批准 |
| 交接 | 固定进入 `PENDING_LEGAL_REVIEW` | 在本项目内升级为“可以提交” |

## 6. 成功标准

当前版本成功，不是指案件分析正确，而是同时满足：

- 复核者能够重现扫描器实际读取了哪些字节；
- 结构化记录不存在未报告的悬空或重复标识符；
- 清单、依赖、文档和案件包快照与当前输入一致；
- 相关输入变化后旧快照不能继续通过；
- 报告明确区分已验证的技术属性与未验证的法律属性；
- 无法由当前项目证明的内容不会获得提交或法律正确性状态。

## 7. 重新定位触发条件

只有 [v1.0 生产门禁](https://github.com/f12336414-ship-it/labor-arbitration-skill/milestone/1) 中对应的法律内容、身份审批、隐私安全和真实案件评测完成并获得相应负责人批准后，产品目的才能扩展到真实案件法律判断或提交辅助。单纯增加提示词、示例规则或免责声明不能改变当前定位。
